## 1. 系統角色與入口

| 角色        | 路由前綴       | Layout 元件              | 主色調       |
|-------------|---------------|--------------------------|-------------|
| 訪客（找房） | `/explore`    | 無（獨立頁面）            | 藍色         |
| 未登入       | `/`           | 無（全螢幕）              | —           |
| 房東         | `/landlord/`  | `LandlordLayout.vue`     | 金色/靛色    |
| 租客         | `/tenant/`    | `TenantLayout.vue`       | 綠/藍色      |
| 管理員       | `/admin/`     | `SuperAdminLayout.vue`   | 深色＋紅色   |

**入口選擇頁** (`/`)：使用者選擇「房東」或「租客」身份後進入 Login 頁面。頁面底部提供「瀏覽空置房間（無需登入）」連結至 `/explore`。

**兩條截然不同的路線：**
```
訪客（找房）
  → /explore（無需登入）→ 找到房間 → 聯絡房東 → 取得代碼 → /register → /onboarding

確認房客（有代碼）
  → /register → /onboarding（必填代碼）→ tenant portal
```

## 2. 認證與新手引導流程

```
[入口頁 /]
   ↓ 選擇身份
[登入 /login]  ←→  [註冊 /register]
   ↓ 登入成功
[新手引導 /onboarding]  ← 僅首次，尚無 profile
   ↓ 填完 profile
[角色儀表板]
```

### 2.1 登入頁 (`/login`)
- 支援 Email/密碼 與 Google 登入
- Google 登入在手機瀏覽器使用 redirect 模式（非 popup）
- 登入後判斷是否有 profile：無 → `/onboarding`；有 → 對應 dashboard
- 返回按鈕回到身份選擇頁

### 2.2 註冊頁 (`/register`)
- Email 格式驗證 + 密碼最少 6 字元 + 確認密碼一致
- 即時錯誤顯示（非 submit 後才顯示）
- 成功後導向 `/onboarding`

### 2.3 新手引導 (`/onboarding`)
- 必填：姓名、手機（格式 09XXXXXXXX）
- 房東：系統自動產生 8 碼 `landlordCode`
- 租客：**landlordCode 為必填**（無「稍後綁定」選項）
  - 輸入時 debounce 500ms 後自動查詢 Firestore
  - 即時顯示「✓ 找到房東：{name}」或「找不到此代碼」
  - 輸入框右側顯示驗證狀態 icon（loading spinner / check / error）
  - 提交按鈕在 `landlordFound === false` 時 disabled
  - 頁面底部顯示「還沒確定？先去瀏覽找房頁」連結
- 完成後導向對應 dashboard

> **設計原則**：走到 onboarding 的租客理應已拿到代碼；未拿到代碼者應先從 `/explore` 開始。

## 3. 公開找房頁 (`/explore`)

**設計定位**：類租屋網站的公開瀏覽體驗，任何人不需登入即可訪問。

### 頁面結構

| 區塊 | 說明 |
|------|------|
| Header | Logo、「登入」、「已有代碼？立即註冊」CTA |
| Hero Banner | 頁面說明 + 引導說明（找到喜歡 → 聯繫房東 → 取得代碼 → 註冊） |
| 篩選列（sticky） | 地區搜尋、格局選單、價格區間選單、結果數 |
| 房間卡片 Grid | 1/2/3 欄響應式 |
| Footer CTA | 「已談好？完成租客註冊」按鈕 |

### 資料來源
- 查詢條件：`isPublic == true` AND `status == 'vacant'`
- 每筆 room 文件包含 `landlordName`、`landlordPhone`（房東儲存時 denormalize 寫入）
- 每筆 room 另批次查詢 `public_profiles/{landlordId}` 取得 `lineBotId`、`avgRating`、`reviewCount`

### 房間卡片功能
- 封面照片（hover 放大效果）
- 格局標籤 overlay（左下）
- **照片數 overlay（右下）→ 點擊開啟照片 Lightbox（全屏放大，支援左右切換 + 縮圖列）**
- 房源名稱、月租金
- **地址隱私**：只顯示城市+行政區（如「台北市大安區」），點擊開啟 Google Maps 新分頁（完整地址搜尋）
- 規格 chips：坪數、格局
- 房東姓名（頭像以姓名首字顯示）+ 評分星數
- 「電話」按鈕 → 點擊後顯示號碼（防爬蟲）
- 「LINE 詢問」按鈕（如房東有設定 LINE Bot）

### 房東個人頁 (`/explore/landlord/:landlordId`)
- 從 `public_profiles` 讀取：`displayName`、`name`（fallback）、`description`、`lineBotId`
- 顯示評分、評價列表（`isVisible == true` 的評價）
- 撰寫評價表單：姓名（暱稱可）、星評 1–5、內容
  - 評價提交由前端 `addDoc(reviews)` 完成（不需登入）
  - `public_profiles.avgRating` / `reviewCount` 由 Cloud Function `onReviewCreated` 自動更新（不在前端更新）

### UX 規則
- 「電話」按鈕點一下才顯示電話（防爬蟲、避免直接曝光）
- 載入時顯示 Skeleton 卡片（6 張）
- 無結果時顯示空白狀態 + 建議調整篩選
- 篩選變更即時重新過濾（前端過濾，無需重新查詢）
- `/explore` 與 `/explore/landlord/:id` 不受 router guard 攔截（已登入者也可訪問）

### isPublic 欄位（房東端）
- 只有 `status === 'vacant'` 的房間可以開啟 `isPublic`
- 狀態改為 `occupied` 或 `maintenance` 時，`isPublic` 自動設為 `false`
- 房間卡片顯示「公開刊登」綠色小徽章（右上角）
- Modal 中的 toggle 在非空置狀態時顯示 disabled

## 4. 房東 UX 流程

### 導覽結構（側邊欄，15 項）

```
儀表板 → 房源管理 → 租客列表 → 社區公告 → 帳務管理
→ 投資試算 → 電表登錄 → 電表歷史 → 報修管理 → 大樓資訊
→ 電子合約 → 收據產生 → 評價管理 → 訊息中心 → 系統設定
```

通知徽章顯示於：**訊息中心**（未讀數）、**報修管理**（待處理件數）、**租客列表**（逾期未繳數）

---

### 4.1 儀表板 (`/landlord/dashboard`)

**目的**：一覽物業狀態、財務概況、快速操作

**區塊**：
- 統計卡片：房間總數、已入住 / 空置 / 維修中、未付 / 已付 / 逾期金額
- 快速電表登錄：顯示前 5 間房間，可即時輸入讀數
- 最新報修：最近 3 筆（待處理優先）
- 快捷操作：新增房間按鈕

**UX 規則**：
- 統計卡片數字點擊後跳轉對應管理頁並自動套用篩選
- 快速電表儲存需顯示成功 Toast，不重新整理整頁
- 報修卡片狀態顏色：待處理=紅、處理中=橙、完成=綠

---

### 4.2 房源管理 (`/landlord/rooms`)

**目的**：新增、編輯、刪除租賃物件

**版面**：卡片網格（3 欄，響應式）

**每張卡片顯示**：封面圖、房間名稱、狀態徽章、房型標籤、金額、地址、現住租客姓名

**篩選**：全部 / 已入住 / 空置 / 維修中

**Modal（新增/編輯）規則**：
- 照片：拖曳上傳，1–10 張，可設定封面
- 房型選項：獨立套房 / 分租套房 / 雅房 / 整層住家
- 狀態為「已入住」時才顯示租客資訊欄位
- 儲存成功後卡片即時更新，無需重新整理
- 儲存時自動 denormalize `landlordName`、`landlordPhone` 至 room 文件（供找房頁使用）

---

### 4.3 租客列表 (`/landlord/tenants`)

**目的**：管理所有租約關係

**狀態邏輯**：
- `已綁定帳號`：租客已在系統內註冊並綁定，顯示藍色徽章
- 付款狀態：正常 / 未付款 / 逾期 / 待確認

**統計列**：總租客數、即將到期（60 天內）、逾期未繳數

**篩選**：全部 / 付款正常 / 有問題（逾期＋未付）

**Modal 規則**：
- 選擇房間後自動帶入租金
- 到期日若距今 ≤ 60 天，顯示警示 icon
- 新增租客若選擇已有帳號者，同步建立 contracts 文件

---

### 4.4 社區公告 (`/landlord/announcements`)

**類別**：一般通知 / 重要公告 / 維修通知 / 活動訊息

**UX 規則**：
- 置頂公告排序優先，置頂最多顯示 3 則（超過則提示）
- 發布即推播通知給所有已綁定租客
- 檢視數即時累加（租客每次開啟 +1）

---

### 4.5 帳務管理 (`/landlord/financials`)

**月份切換**：上/下月箭頭，預設當月

**主要 Tab**：收入 / 支出 / 歷史

**計算模式**：
- `bill_share`（分攤制）：輸入電費總額 → 系統依各戶用電量比例分攤
- `tiered`（階梯式）：依台電階梯電價逐戶計算，支援夏月/非夏月/平均/自動

**UX 規則**：
- 「自動產生帳單」按鈕需彈出確認對話框（不可誤觸）
- 匯入台電帳單前，需先完成電表登錄
- 負盈餘（支出 > 收入）時淨利顯示紅色

---

### 4.6 投資試算 (`/landlord/investment`)

**目的**：計算投報率、預期收益、稅務成本

**UX 規則**：
- 試算結果即時更新（無需按計算按鈕）
- 數字以千分位格式顯示

---

### 4.7 電表登錄 (`/landlord/meter-reading`)

**目的**：批量登錄各房間電錶讀數並計算電費

**功能**：
- 手動逐筆輸入
- Excel 匯入（依 `租屋資料彙整.xlsm` 格式）
- 即時顯示用電量 (kWh) 與預估費用
- 設定 Modal：計費模式、費率參數

**UX 規則**：
- 讀數若小於上期讀數，顯示紅色警示
- 每次儲存在 `meter_readings` 新增一筆紀錄
- 批次儲存成功後顯示摘要（X 間房已更新）

---

### 4.8 電表歷史 (`/landlord/meter-history`)

**篩選**：月份 × 房間
**欄位**：日期、房間、上期讀數、本期讀數、用電量、費用
**唯讀**：此頁不允許編輯

---

### 4.9 報修管理 (`/landlord/repairs`)

**優先級顏色**：高=紅、中=橙、低=藍

**狀態流轉**：
```
待處理 → 處理中 → 已完成
```

**UX 規則**：
- 狀態變更即時同步租客端（租客可看到進度更新）
- 完成後可輸入維修費用（記入財務支出）
- 側邊欄徽章數 = 「待處理」件數

---

### 4.10 大樓資訊 (`/landlord/building-info`)

**內容**：逃生圖、設施位置、使用規範、注意事項

**UX 規則**：
- 可上傳圖片（逃生圖等）
- 租客端同步顯示（唯讀）

---

### 4.11 電子合約 (`/landlord/contract`)

**兩步驟流程**：
1. 填寫合約資訊（房間、租客、租期、金額、簽名）
2. 預覽 PDF → 確認無誤 → 下載

**UX 規則**：
- 選擇房間後自動帶入地址與租金
- 簽名欄為必填（Canvas 手寫簽名元件）
- 下載前必須完成步驟 2 預覽確認

---

### 4.12 收據產生 (`/landlord/receipts`)

**類型**：租金收據 / 押金收據

**UX 規則**：
- 選擇房間後自動帶入地址與租客姓名
- 可直接下載 PDF，無需儲存至雲端

---

### 4.13 評價管理 (`/landlord/reviews`)

**目的**：查看租客對自己的評價，可回覆或隱藏

**UX 規則**：
- 房東可隱藏不實或不當評價（`isVisible` 設為 `false`）
- 房東可針對每則評價回覆一次（`landlordReply` 欄位）
- 隱藏的評價在租客端不顯示，但管理員可見
- 平均評分與評價數由 Cloud Function `onReviewCreated` 自動更新至 `public_profiles`，**不在前端寫入**

---

### 4.14 訊息中心 (`/landlord/messages`)

**Tab**：全部 / 未讀 / 已回覆

**來源徽章**：LINE（綠）/ Web（灰）

**UX 規則**：
- 開啟訊息對話後，該訊息自動標記為已讀
- 回覆送出後，訊息移入「已回覆」tab
- 側邊欄徽章 = 未讀數

---

### 4.15 系統設定 (`/landlord/settings`)

**區塊**：
- 基本資訊（姓名、手機、Email — Email 唯讀）
  - 儲存時同步更新 `public_profiles.displayName` 與 `public_profiles.name`（供找房頁顯示）
- 預設匯款帳戶（銀行代號、帳號、戶名）
- LINE Bot 設定（Channel Secret、Access Token）
  - 儲存後同步 `lineBotId` 至 `public_profiles`

## 5. 租客 UX 流程

### 導覽結構（側邊欄，6 項）

```
儀表板 → 我的帳單 → 社區公告 → 報修申請 → 聯繫房東 → 大樓資訊
```

通知指示：**聯繫房東** 顯示「有新回覆」脈衝點

---

### 5.1 儀表板 (`/tenant/dashboard`)

**區塊**：
- 左：租約資訊卡（房間號、租期、月租、繳款日）
- 右：本月帳單摘要卡（費用明細、總計、繳款按鈕）
- 若無合約：顯示引導訊息（聯繫房東完成綁定）

**UX 規則**：
- 繳款按鈕點擊後顯示確認對話框（金額 + 確認/取消）
- 帳單已繳則隱藏繳款按鈕，顯示「已繳清」狀態
- 租約到期日 ≤ 30 天顯示橙色警示

---

### 5.2 我的帳單 (`/tenant/bills`)

**Tab**：未繳 / 已繳

**摘要卡**：總未繳金額、未繳筆數、上次繳款日

**UX 規則**：
- 逾期帳單（超過到期日）以紅色標示
- 點擊帳單列可展開明細（租金 / 電費 / 其他）
- 已繳帳單可下載收據 PDF

---

### 5.3 社區公告 (`/tenant/announcements`)

**唯讀**：租客只能瀏覽，不可發文

**UX 規則**：
- 置頂公告排在最上方，帶有「重要」標籤
- 開啟公告即累加檢視數（房東端可見）
- 未讀公告在側邊欄以點標示（選配）

---

### 5.4 報修申請 (`/tenant/repairs`)

**狀態查看**：
```
待處理（紅）→ 處理中（橙）→ 已完成（綠）
```

**新增報修 Modal**：
- 房間號自動帶入（來自合約）
- 類別：水電 / 家電 / 結構 / 傢俱 / 其他
- 標題 + 詳細描述
- 提交後狀態預設「待處理」

**UX 規則**：
- 已完成工單不可再編輯
- 每筆工單顯示提交時間與最後更新時間

---

### 5.5 聯繫房東 (`/tenant/contact`)

**左側**：房東資訊卡（姓名、電話可撥打、Email 可複製、LINE ID 可複製）

**LINE Bot 綁定**：
- 顯示綁定狀態（已綁定 / 未綁定）
- 未綁定：顯示綁定說明
- 已綁定：顯示解除綁定按鈕

**右側**：訊息對話（類 chat UI）
- 最新訊息在下方
- 回覆輸入框 + 送出按鈕
- 有新回覆時顯示「新回覆」脈衝點（側邊欄亦同步）

**UX 規則**：
- 若無綁定房東，整頁顯示引導訊息（提示輸入房東代碼）
- 訊息來源 LINE/Web 用徽章區分

---

### 5.6 大樓資訊 (`/tenant/building-info`)

**唯讀**：查看房東上傳的逃生圖、設施位置、使用規範

## 6. 管理員 UX 流程

### 導覽結構（深色側邊欄，5 項）

```
系統總覽 → 房東管理 → 租客與配對 → 資料庫操作 → 系統模擬器
```

---

### 6.1 系統總覽 (`/admin/dashboard`)

**統計**：總房東數、總租客數、未配對租客數（紅色醒目）

**快捷卡片**：直接跳轉各管理頁

---

### 6.2 房東管理 (`/admin/landlords`)

**操作**：搜尋、分頁、模擬登入（Impersonate）、同步、刪除

**模擬登入規則**：
- 進入模擬模式後，頂端顯示「目前模擬：{房東名稱}」警示列
- 任何寫入操作使用該房東的 `effectiveUid`
- 側邊欄顯示「結束模擬」按鈕，點擊後恢復管理員身份
- 房東側邊欄登出按鈕改為「結束模擬」

---

### 6.3 租客與配對 (`/admin/tenants`)

**統計**：總租客、已配對、未配對（紅色警示）

**指派房東 Modal**：
- 搜尋房東（姓名或代碼）
- 選擇後確認寫入 `boundLandlordCode`

**UX 規則**：
- 未配對租客在列表頂端優先顯示
- 刪除租客需二次確認（輸入「確認刪除」文字）

---

### 6.4 資料庫操作 (`/admin/database`)

**高風險操作規則**：
- 批量刪除需輸入確認文字
- 所有操作需顯示影響筆數預覽後才能執行
- 操作結果記錄至 admin_logs（含 uid、時間、操作類型）

---

### 6.5 系統模擬器 (`/admin/simulator`)

**用途**：測試使用者流程、帳單計算、電表邏輯

**UX 規則**：
- 模擬操作不寫入正式資料（使用 sandbox collection 或 dry-run 模式）
- 顯示每步驟的計算過程與中間值

## 7. 共用 UX 規則

### 7.1 Toast 通知

| 場景             | 類型    | 持續時間 |
|------------------|---------|---------|
| 儲存成功          | success | 3 秒    |
| 操作失敗/錯誤     | error   | 5 秒    |
| 警示（如到期提醒） | warning | 4 秒    |
| 一般資訊          | info    | 3 秒    |

- 使用 `ToastContainer.vue`，位置：右下角
- 同時最多顯示 3 則，超過則佇列等待

### 7.2 Modal 規則

- 開啟 Modal 時背景加深遮罩（`bg-black/50`）
- ESC 鍵關閉 Modal（需先確認有無未儲存變更）
- 確認刪除的 Modal 需使用紅色確認按鈕
- 必填欄位標示 `*`，submit 時統一驗證並 focus 第一個錯誤欄位

### 7.3 載入狀態

- 資料載入中：顯示 Skeleton 或 Spinner
- 按鈕操作中：按鈕 disabled + 顯示 loading icon，防止重複觸發
- 非同步操作失敗：顯示 error Toast，按鈕恢復正常

### 7.4 響應式設計

- 斷點：`lg` (1024px) 以上固定顯示側邊欄；以下改為 Hamburger 選單
- 表格在手機版改為卡片列表
- Modal 在手機版為底部滑入或全螢幕

### 7.5 表單驗證規則

| 欄位      | 規則                         |
|-----------|------------------------------|
| 手機      | 必須符合 `09XXXXXXXX`         |
| Email     | 標準格式驗證                  |
| 密碼      | 最少 6 字元                   |
| 金額      | 正整數，不得為 0              |
| 日期區間   | 結束日 > 開始日               |

### 7.6 空白狀態

- 每個清單/表格若無資料，顯示說明文字 + 引導操作按鈕（例如「尚無房間，點此新增」）
- 不顯示空白表格框架

### 7.7 角色路由保護

- Router guard 依 `meta.role` 驗證，不符合角色導回對應 dashboard
- 未登入訪問任何受保護路由，導向 `/login`
- 管理員可訪問所有路由（含模擬登入時的房東路由）
- `/explore`、`/explore/landlord/:id` 永遠可訪問，不受 guard 攔截

## 8. 資料流與權限邊界

### Firestore 查詢過濾規則

| Collection        | 房東查詢條件          | 租客查詢條件        | 管理員 |
|-------------------|----------------------|--------------------| -------|
| rooms             | `landlordId == uid`  | —（透過合約）       | 全部   |
| rooms（公開）     | —                    | `isPublic==true && status==vacant`（任何人）| 全部 |
| tenants           | `landlordId == uid`  | —                  | 全部   |
| contracts         | `landlordId == uid`  | `tenantId == uid`  | 全部   |
| bills             | `landlordId == uid`  | `tenantId == uid`  | 全部   |
| repair_requests   | `landlordId == uid`  | `tenantId == uid`  | 全部   |
| announcements     | `landlordId == uid`  | 透過 `boundLandlordCode` | 全部 |
| messages          | `landlordId == uid`  | `tenantId == uid`  | 全部   |
| meter_readings    | `landlordId == uid`  | —                  | 全部   |
| reviews           | `landlordId == uid`  | —（公開可讀）       | 全部   |
| public_profiles   | 自己的文件            | —（公開可讀）       | 全部   |

### 關鍵欄位（rooms）

| 欄位 | 說明 |
|------|------|
| `isPublic` | boolean，控制是否顯示於 `/explore` 找房頁 |
| `landlordName` | 房東姓名（儲存時 denormalize，避免找房頁需 join） |
| `landlordPhone` | 房東電話（同上，僅對點擊「電話」的訪客顯示） |

### 關鍵 ID 欄位

- `landlordCode`：8 碼隨機字串，房東專用邀請碼
- `tenantId`：10 碼隨機字串，租客唯一識別
- `boundLandlordCode`：存於租客 profile，連結所屬房東
- `effectiveUid`：管理員模擬時使用，覆蓋實際 uid

### public_profiles 欄位

| 欄位 | 說明 | 寫入時機 |
|------|------|---------|
| `displayName` | 房東顯示名稱 | 房東儲存設定時 |
| `name` | 房東名稱（fallback） | 房東儲存設定時 |
| `description` | 房東自介 | 房東儲存設定時 |
| `lineBotId` | LINE Bot ID | 房東設定 LINE 時 |
| `avgRating` | 平均評分 | Cloud Function `onReviewCreated` 自動更新 |
| `reviewCount` | 評價數 | Cloud Function `onReviewCreated` 自動更新 |

### 寫入權限邊界

- 租客**不可**修改帳單金額、公告內容、房間資料
- 房東**不可**讀取其他房東的資料
- 管理員寫入操作應記錄至 `admin_logs`
- `public_profiles.avgRating` 與 `reviewCount` 只由 Cloud Function 寫入，前端不直接更新
